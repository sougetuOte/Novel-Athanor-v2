# Ship: 変更の棚卸し・論理コミット・後処理

変更ファイルを棚卸しし、論理グループに分けてコミットし、ユーザーへの手動作業を通知する。
`/full-save` のコミット部分を強化した、変更出荷ワークフロー。

引数（任意）: `/ship dry-run`（コミットせず計画のみ表示）

---

## Phase 1: 棚卸し（Inventory）

1. `git status` で全変更ファイルを取得（untracked 含む）
2. `git diff --stat` でステージ済み・未ステージの変更量を確認
3. 各ファイルの変更内容を簡潔に要約（1ファイル1行）
4. 秘密情報チェック: `.env`, `credentials`, `*.key`, `settings.local.json` 等を除外候補に
5. 結果を一覧表示:

```
--- Ship: 棚卸し ---
変更: X files | 新規: X files | 削除: X files
除外候補: [秘密情報ファイルがあれば列挙]
```

## Phase 1.5: ドキュメント同期（Doc Sync）

Phase 1 の棚卸し結果を踏まえ、以下のドキュメントを現状と照合する:

| ファイル | 更新条件 | 更新内容 |
|---------|---------|---------|
| `SESSION_STATE.md` | **毎回** | 完了タスク・進行中タスク・次ステップを実態に更新 |
| `CHEATSHEET.md` | 新コマンド・ルール・スキル追加時 | 該当テーブルに追記 |
| `README.md` | 新モジュール追加・フェーズ進行時 | Phase 進捗セクションの更新 |

### 出力

```
--- Doc Sync ---
SESSION_STATE.md: 更新 / スキップ
CHEATSHEET.md: 更新 / スキップ
README.md: 更新 / スキップ
```

ドキュメント更新があった場合、Phase 2 のグループ分けに `docs` グループとして含める。

## Phase 2: 論理グループ分け（Grouping）

変更を以下の基準で論理グループに分類（最大5グループ）:

### 分類基準
- **同じ目的** の変更をまとめる（例: 「L4 Phase B 実装」）
- **同じレイヤー** の変更をまとめる（例: 「テスト追加」）
- **依存関係** があるファイルは同じグループ（例: 仕様書 + それに基づく実装）

### グループテーブルを表示

```
| Group | 目的 | ファイル数 | ファイル一覧 |
|-------|------|-----------|-------------|
| A | [目的] | N | file1, file2, ... |
| B | [目的] | N | file3, file4, ... |
```

### 判断に迷うファイルの扱い
- 複数グループにまたがるファイル → 主要な目的のグループに入れる
- どこにも属さないファイル → 「chore: その他」グループ

## Phase 3: コミット計画（Plan）

各グループのコミットメッセージ案を生成:

```
| Group | コミットメッセージ | prefix |
|-------|-------------------|--------|
| A | feat(l4): add Ghost Writer formatter | feat |
| B | docs: update SESSION_STATE | docs |
```

### prefix 規則
- `feat`: 新機能
- `fix`: バグ修正
- `docs`: ドキュメント変更
- `chore`: 設定・ツール・保守作業
- `refactor`: リファクタリング
- `test`: テスト

### 停止判断基準
以下に該当するグループは **コミットしない** でユーザーに報告:
- 変更が中途半端（WIP 状態が明らか）
- 仕様書と実装の整合性が取れていない
- テストが必要だが未実行
- 秘密情報が含まれる

## Phase 4: ユーザー確認

コミット計画をユーザーに提示し、承認を求める:

```
--- Ship: コミット計画 ---
[グループテーブル + コミットメッセージ]

実行しますか？
1. 全グループをコミット
2. グループを選択してコミット
3. dry-run（何もしない）
```

引数に `dry-run` が指定されていた場合は Phase 4 で停止。

## Phase 5: コミット実行（Execute）

ユーザー承認後:
1. グループ順に `git add [files]` + `git commit`
2. 各コミット完了を報告
3. **push はしない**（push は `/full-save` または手動で行う）

## Phase 6: 手動作業通知（Manual Actions）

コミット後、以下をチェックして通知:

### 6.1 手動削除すべきファイル
プロジェクトに不要と判断されるファイルを列挙:
- 一時ファイル、デバッグ用の残骸
- `.gitignore` に追加すべきパターン

```
--- 手動削除候補 ---
- [ファイルパス]: [理由]
AI は削除を実行しません。
```

### 6.2 ユーザーに委ねる作業

```
--- ユーザー作業 ---
- [ ] git push（リモートに反映）
- [ ] [その他必要な作業]
```

## Phase 7: 完了報告

```
--- Ship 完了 ---
コミット: X 件（Group A, B, ...）
スキップ: X 件（理由: ...）
手動削除候補: X 件
ユーザー作業: X 件

次のステップ:
  git push          （リモートに反映）
  /full-save        （SESSION_STATE + push + daily）
  /quick-save       （セッション状態のみ保存）
```
