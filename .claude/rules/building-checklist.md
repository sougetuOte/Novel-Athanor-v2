# BUILDING 品質チェックリスト

Phase 1 監査で発見された不具合パターンを防止するルール。
TDD サイクルの各ステップで適用する。

## Red（テスト作成時）

- **R-4: FR チェックリスト駆動テスト**
  タスク定義の FR 番号一覧をテストファイル docstring に転記する。
  各 FR に最低 1 テストが対応することを確認してから実装に進む。

- **R-5: 異常系テストの義務**
  正常系だけでなく、エラーパス・境界値・例外ケースのテストも Red で書く。
  「try/except があるなら、except に落ちるテストがある」を原則とする。

- **R-12: テスト衛生**
  - テストメソッドには `-> None` 型注釈を付ける。
  - 遅い・不安定なテストには `@pytest.mark.slow` を付け、`pyproject.toml` にマーカー登録する。
  - 同じヘルパーを3テストファイル以上で使う場合は `conftest.py` に集約する。
  - import は全てファイル先頭に置く（テスト内のインライン import 禁止）。

## Green（実装時）

- **R-2: 有限セットは dict ディスパッチ**
  3 分岐以上の有限値セット（purpose, state, type 等）は if-chain ではなく dict/map で処理する。
  未登録値は KeyError で明示的に失敗させる。

- **R-6: else のデフォルト値禁止**
  if-elif-else や dict.get() のデフォルト値を書く際、
  「この else/default に到達する正当な入力」を言語化できなければ `raise ValueError` にする。
  **特に `dict.get(key, default)` は要注意**: Enum/有限セットのキーなら `dict[key]` を使い、
  KeyError で未知の値を即座に検出する。

- **R-3: 定数定義 → 使用の即時接続**
  定数・frozenset・Enum を定義したら、同じサイクル内でその定数を参照するコードを書く。
  「後で使う予定」の未参照定数を残さない。

- **R-7: スレッド安全性**
  `concurrent.futures` 等で並列処理を行う場合:
  - 共有オブジェクトを複数スレッドから**書き込まない**。
  - 各スレッドにローカルな結果オブジェクトを持たせ、メインスレッドでマージする。
  - `list.append` や `dict.update` でも競合するため、ロックなしの共有変更は禁止。

- **R-8: ミュータブル引数の保護**
  リストや辞書を引数に受け取る関数で、その引数を変更する可能性がある場合:
  - `forbidden_keywords = list(forbidden_keywords) if forbidden_keywords else []` のように防御コピーする。
  - `forbidden_keywords = forbidden_keywords or []` は**禁止**（呼び出し側のリストを破壊する）。
  - デフォルト引数に `[]` や `{}` を使わない（`None` にして関数内で生成）。

- **R-9: None vs falsy の区別**
  `Optional[T]` な値を条件判定する際:
  - **`if x is not None:`** を使う（`if x:` は禁止）。
  - 空文字列 `""`, 空リスト `[]`, `0` は falsy だが None ではない。
  - 特に merge/update 系の処理で「値が設定されていない」と「空値が設定された」を区別する。

- **R-10: モジュール横断 DRY**
  同じロジックが **2ファイル以上** に出現したら共通ユーティリティに抽出する。
  （03_QUALITY_STANDARDS の「3回ルール」より厳しい基準: パーサー等の定型処理は2回で抽出）
  抽出先は `_utils.py` や `_common.py` 等、パッケージ内のプライベートモジュールとする。

- **R-11: 型の精度**
  取りうる値が有限集合の場合、`str` ではなく `Literal["a", "b"]` を使う。
  mypy が未知の値を静的に検出できるようにする。

## Green 直後（実装完了時）

- **R-1: FR 突合チェック**
  テスト通過後、対応する FR/設計仕様のセクションを**再読**し、
  フィールド名・定数値・メッセージ文言が**文字単位で一致する**か照合する。

- **R-5 続: カバレッジ確認**
  `pytest --cov` で未カバー行を確認する。
  未カバーが「テスト不要（GUI イベント等）」か「テスト必要（エラーパス等）」かを判断し、
  必要なら追加テストを書く。目標: 各モジュール 90% 以上。

- **S-1: 仕様同期チェック**（`spec-sync.md` 参照）
  新しい定数・メソッド・スキーマを追加した場合、`docs/specs/` の対応セクションに
  記載があるか確認する。記載がなければ同サイクル内で仕様を更新する。

- **S-2: `__init__.py` 公開 API 確認**
  新しいパッケージやモジュールを追加した場合、`__init__.py` に `__all__` を定義し、
  公開クラス・関数を明示的にエクスポートする。
