# 監査修正ポリシー

## 目的

レビュー・リファクタリング時に発見された Issue について、
対応可能なものはすべて同セッション内で修正することを義務化する。

## ルール

### A-1: 全重篤度への対応義務

監査レポートの **Critical / Warning / Info** すべての Issue について、
現フェーズで対応可能かどうかを判断し、対応可能なものは**すべて修正する**。
「Warning だから後回し」「Info だから無視」は禁止。

### A-2: 対応不可 Issue の明示

対応不可と判断した場合は、以下を明記する:
- **理由**: なぜ今対応できないか（Phase 2 依存、外部 API 未実装等）
- **追跡方法**: どこに記録するか（`docs/tasks/` or Issue tracker）
- **暫定対策**: あれば記載

### A-3: 修正後の再検証

修正した Issue については、以下を実施する:
1. テスト追加（テストで検証可能な場合）
2. ruff check でリント通過確認
3. 全テスト実行で回帰なし確認

### A-4: 仕様ズレの同時修正

コード修正に伴い仕様書との不整合が発生する場合、
コードと仕様書を同一サイクル内で同時更新する（Atomic Commit 原則）。

### A-5: 仕様突合の義務（S-1 監査版）

監査時に以下を全件チェックする:
- `docs/specs/` のデータモデル定義が実装と一致するか（フィールド名、型、Enum値）
- `docs/specs/` のアーキテクチャ図・ディレクトリ構造が現状と一致するか
- Agent 定義（`.claude/agents/`）が実装状況と整合するか

> **教訓**: BUILDING 中の S-1 だけでは仕様ドリフトを完全に防げない。
> 監査フェーズでの突合を A-5 として義務化する。

### A-6: 既存ルール適用漏れの点検

監査時、`building-checklist.md` の全ルール（R-1〜R-12）について、
対象コードベースに違反が残っていないかを **grep/検索ベース** で確認する。
目視だけの確認は禁止。

> **教訓**: R-6（else デフォルト値禁止）は BUILDING 時に制定済みだったが、
> 5箇所以上で適用漏れが残存していた。ルールの存在と適用は別問題。

## 監査レポート完了条件

```
対応可能な Issue: X件中 X件修正済み
対応不可の Issue: X件（理由と追跡先を記録済み）
テスト結果: 全件 PASSED
ruff: All checks passed
mypy: Success
```

上記を満たさない限り、監査完了とはしない。
